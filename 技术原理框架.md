## 一、项目概述

本项目是一个面向耐辐射奇球菌（*Deinococcus radiodurans*）研究领域的学术文献检索 Web 应用。它整合了从 PubMed 获取的 1,344 篇规范化文献数据，支持用户通过**序号、标题关键词、作者名或 PMID** 进行多维度检索，并在前端以卡片式布局和模态框详情页的形式展示结果。

项目采用**双运行模式**设计：
- **推荐模式**：`main.py` — 纯前端单页应用（SPA），通过 Python 内置 HTTP 服务器一键启动，无需任何第三方依赖
- **备用模式**：`app.py` — Flask 后端驱动的全栈应用，提供 RESTful API，适合需要后端处理能力的场景

```

[index.html] 加载 → <script> 标签执行 JS

### 4.2 搜索请求数据流（main.py 模式）

```
[用户在输入框键入查询] → [点击搜索或按 Enter]
    │
    ↓ 触发 render() 函数
[前端 search() 函数] 在内存 DB 中执行匹配：
    │
    ├── 若输入纯数字且 DB[query] 存在 → 精确返回 1 条
    │
    ├── 否则按标题关键词匹配（遍历所有条目，小写包含判断）
    │   └─ 最多返回 30 条
    │
    ├── 标题无结果 → 按作者名匹配（同上逻辑）
    │   └─ 最多返回 30 条
    │
    ├── 作者无结果 → 按 PMID 精确匹配
    │   └─ 返回 1 条
    │
    └── 仍无结果 → 按摘要全文关键词匹配
        └─ 最多返回 30 条
    │
    ↓ 匹配结果数组

    ├─ 无结果 → 显示 "未找到与 xxx 相关的文献"
    └─ 有结果 → 生成结果卡片 HTML（含关键词高亮 <mark>）
    │
    ↓ DOM 更新
[用户看到结果列表] → 点击卡片
    │
    ↓ 触发 show(entry)
[模态框弹出] → 显示文献详情：
    ├─ 标题、作者、期刊、年份
    ├─ PMID / PMCID / DOI（外链至 PubMed / DOI.org）

    └─ MeSH Terms 标签云

### 4.3 搜索请求数据流（app.py 模式）

```
[用户输入查询] → [前端 JS 发送 fetch('/api/search?q=xxx')]
    │
    ↓ HTTP GET /api/search?q=xxx
[Flask 路由处理] → [api_search() 函数]
    │
    ↓ 后端执行相同匹配逻辑
[后端搜索算法]：
    ├─ 纯数字 → 先尝试序号精确匹配
    ├─ 标题关键词匹配（前 20 条）

```

## 五、核心代码逻辑解析

### 5.1 前端搜索引擎（index.html → JavaScript）

```javascript
function search(query) {
    if (!DB || !query) return [];
    const q = query.toLowerCase(), r = [];

    // 第一层：精确序号匹配（优先）
    if (/^\d+$/.test(q) && DB[q]) return [DB[q]];

    // 第二层：标题关键词（最多 30 条）
    for (const k in DB) {
        if (DB[k].title.toLowerCase().includes(q)) {
            r.push(DB[k]);
            if (r.length >= 30) break;
        }
    }
    if (r.length) return r;

    // 第三层：作者名匹配（最多 30 条）
    for (const k in DB) {
        if (DB[k].authors.toLowerCase().includes(q)) {
            r.push(DB[k]);
            if (r.length >= 30) break;
        }
    }
    if (r.length) return r;

    // 第四层：PMID 精确匹配
    for (const k in DB) {
        if (DB[k].pmid === query) return [DB[k]];
    }

    // 第五层：摘要全文关键词（最多 30 条）
    for (const k in DB) {
        if (DB[k].abstract.toLowerCase().includes(q)) {
            r.push(DB[k]);
            if (r.length >= 30) break;
        }
    }
    return r;
}
```

**设计要点**：
- 搜索优先级严格分层，确保最常见的查询（序号、标题）最先命中
- 结果上限 30 条，防止大关键词（如 "Deinococcus"）导致渲染卡顿
- 全小写转换（`toLowerCase()`）实现大小写不敏感匹配
- 无模糊匹配 / 无全文索引 / 无相关性排序 —— 纯线性遍历，依赖数据规模可控（1,344 条）

### 5.2 前端渲染引擎（关键词高亮）

```javascript
const hl = (t, q) => {
    if (!q) return esc(t);
    // 转义正则特殊字符，防止注入
    const r = new RegExp('(' + q.replace(/[.*+?^${}()|[\]\\]/g, '\\$&') + ')', 'gi');
    return esc(t).replace(r, '<mark>$1</mark>');
};
```

**安全设计**：
- `esc()` 函数将 HTML 特殊字符（`& < > " '`）转义为实体，防止 XSS
- 正则表达式前对查询词进行 escape，防止用户输入破坏 RegExp 语法

### 5.4 后端搜索算法（app.py → Python）

```python
@app.route('/api/search')
def api_search():
    query = request.args.get('q', '').strip()
    if not query:
        return jsonify({'error': '请输入查询内容'})

    results = []

    # 尝试按编号精确匹配
    if query.isdigit():
        num = query
        if num in DATABASE:
            results.append(DATABASE[num])

    # 按标题关键词搜索（前20个结果）
    if not results:
        q_lower = query.lower()
        for num, entry in DATABASE.items():
            title = entry.get('title', '') or ''
            if q_lower in title.lower():
                results.append(entry)
            if len(results) >= 20:
                break

    # 按作者搜索
    if not results:
        q_lower = query.lower()
        for num, entry in DATABASE.items():
            authors = entry.get('authors', '') or ''
            if q_lower in authors.lower():
                results.append(entry)
            if len(results) >= 20:
                break

    # 按PMID搜索
    if not results and query.isdigit():
        for num, entry in DATABASE.items():
            if entry.get('pmid') == query:
                results.append(entry)
                break

    return jsonify({'query': query, 'count': len(results), 'results': results})
```

**与前端模式的差异**：
- 后端结果上限为 **20 条**（前端为 30 条）
- 后端缺少**摘要全文检索**层（前端有第 5 层）
- 返回 JSON 含 `query`、`count`、`results` 三字段

### 5.5 一键启动器（main.py）

```python
class Handler(http.server.SimpleHTTPRequestHandler):
    def log_message(self, format, *args):
        pass  # 静默日志，控制台干净

    def end_headers(self):
        self.send_header('Access-Control-Allow-Origin', '*')  # 允许跨域
        super().end_headers()

def open_browser():
    webbrowser.open(f'http://localhost:{PORT}', new=2)  # new=2 = 新标签页

# 启动流程
with socketserver.TCPServer(('', PORT), Handler) as httpd:
    threading.Timer(1.0, open_browser).start()  # 延迟 1 秒开浏览器
    httpd.serve_forever()
```

**关键设计**：
- `socketserver.TCPServer.allow_reuse_address = True`：防止端口被占用后无法立即重启
- `end_headers()` 注入 `Access-Control-Allow-Origin: *`：解决 CORS 问题
- `threading.Timer(1.0, open_browser)`：服务器启动 1 秒后自动打开浏览器，用户体验流畅

## 七、设计决策与权衡

### 7.1 为什么选择纯前端搜索？

| 方案 | 优点 | 缺点 | 本项目选择 |
|------|------|------|-----------|
| 纯前端（当前） | 零后端延迟；可离线使用；部署极简（2 个文件） | 首次加载 2.5 MB JSON；搜索性能受限于客户端 | ✅ **采用** |
| 后端 API 搜索 | 前端体积小；可支持复杂查询 | 每次搜索都有网络延迟；需要维护后端 | ⚪ 备用（app.py） |
| 数据库索引 | 海量数据下性能高 | 引入依赖（SQLite/Elasticsearch） | ❌ 不适用（仅 1,344 条） |

**结论**：1,344 条文献的数据规模极小，纯线性遍历在毫秒级完成，无需引入索引。前端一次性加载后，搜索响应即时，用户体验优于 API 往返。

### 7.2 为什么选择内置 HTTP 服务器？

直接双击打开 `index.html` 会因浏览器的 **CORS（同源策略）** 导致 `fetch('database.json')` 失败。Python 内置 `http.server` 模块提供：
- 同源上下文（`http://localhost:8765`）
- 自定义响应头（注入 `Access-Control-Allow-Origin: *`）
- 零安装成本（Python 标准库）

### 7.3 数据规范化来源

原始数据通过 **PubMed E-utilities API** 的 `efetch.fcgi` 接口获取标准化 XML，解析后生成 `database.json`。相比用户提供的原始文本文件，PubMed API 返回的结构化 XML 避免了：
- 作者信息行缺失
- 日期格式不统一
- 多语言文献的特殊格式
- 摘要和标题的分隔方式不一致
