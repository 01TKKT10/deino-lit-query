# 搜索算法改进说明

## 概述

本次改进将 deino-lit-query 项目的搜索算法升级为 **v4 投票计数 + 唯一性核对** 机制。两个版本（前端 `index.html` 和后端 `app.py`）保持逻辑完全一致。

**核心变更**：将查询拆分为单词后，每个单词独立在全库检索标题匹配，通过投票计数和逐轮唯一性核对来确定最相关的文献。

---

## 算法核心逻辑（v4）

### 1. 单词分割

查询字符串按空格/空白字符分割为单词数组，保留单词内部的标点符号。

| 输入示例 | 分割结果 |
|---------|---------|
| `"deino radiodurans radiation resistance"` | `["deino", "radiodurans", "radiation", "resistance"]` |
| `"radiodurans."` | `["radiodurans."]` |
| `"DNA repair"` | `["DNA", "repair"]` |

### 2. 投票计数 + 唯一性核对

**核心流程：**

1. **每个单词独立在全库检索**标题匹配（不递进过滤，而是独立投票）
2. 维护投票计数器：`Map<number, {entry, count}>`
3. 先处理**前8个单词**（或全部单词，取最小值），统计投票
4. **核对**：最高票数的文献是否**唯一**
   - **唯一**（只有一篇文献票数最高）→ 直接返回该文献
   - **不唯一**（多篇文献并列最高票）→ 继续下一步
5. **增加第9个单词**（如果存在），将其匹配结果加入投票池，重新统计
6. 再次核对最高票是否唯一 → 唯一则返回，否则继续
7. 以此类推，直到用完所有单词或得到唯一解

**示例：**

输入：`"deino radiodurans radiation resistance dna repair protein structure function"`

| 单词 | 匹配文献 | 投票变化 |
|------|---------|---------|
| `"deino"` | {A, B, C, D} | A=1, B=1, C=1, D=1 |
| `"radiodurans"` | {A, B, E} | A=2, B=2, C=1, D=1, E=1 |
| ...处理完8个单词 | ... | A=8, B=7, C=5... |
| **核对** | A是唯一最高(8票) | **→ 返回A** |

**并列最高时的处理：**

如果8个单词后：A=6票，B=6票（并列最高）→ 不唯一 → 增加第9个单词

- 单词9 `"function"` → 匹配到 {A, C} → A=7票，C增加票数
- 再次核对：A=7票，B=6票 → A是唯一最高 → **返回A**

### 3. 边界处理

| 场景 | 处理方式 |
|------|---------|
| 查询少于8个单词 | 直接用全部单词统计投票 |
| 某个单词无匹配 | 不影响投票（该单词不投任何票） |
| 所有单词用完仍不唯一 | 按票数降序排列，返回前30条 |
| 完全没有匹配的单词 | 进入 fallback 链 |

## 保留的原有功能

| 功能 | 触发条件 | 优先级 |
|------|---------|--------|
| **序号精确匹配** | 输入纯数字（如 `189`） | 最高 |
| **标题搜索** | 输入非纯数字时先进行标题投票计数 | 主流程 |
| **作者名搜索** | 标题无结果时搜索作者字段 | Fallback |
| **PMID 精确匹配** | 作者搜索无结果时 | Fallback |
| **摘要全文搜索** | 标题和作者均无结果时 | 最后 Fallback |

## 前后端逻辑一致性说明

两个版本的搜索逻辑完全相同：

1. 都先检查精确序号匹配
2. 都执行投票计数 + 唯一性核对算法
3. 都先处理前8个单词，检查唯一性
4. 不唯一时逐轮增加单词，直到唯一或用完
5. 所有单词用完仍不唯一 → 按票数排序，取前 30 条
6. 都有相同的 Fallback 搜索链（作者 → PMID → 摘要）

## 版本演进

| 版本 | 匹配策略 | 特点 |
|------|---------|------|
| v1 | 简单包含匹配 | 直接检查标题是否包含查询字符串 |
| v2 | 字符递进匹配 | 按字符逐层递进过滤（如 d → de → dei → dein → deino） |
| v3 | 渐进式单词递进 + 相似度评分 | 按单词逐层递进过滤，配合相似度评分排序 |
| v4 | 投票计数 + 唯一性核对 | 每个单词独立投票，通过唯一性核对确定最相关文献 |
| **v5（当前）** | **整词匹配 + 交集 + 子串 fallback** | **使用 `\b` 词边界实现整词匹配，通过交集运算逐步缩小范围；无结果时回退到子串匹配** |
| **v5.1（当前）** | **整词匹配 + 早期返回 + lastNonEmpty 保护** | **在交集构建过程中，每处理一个单词后立即检查唯一性（size==1 立即返回）；单词无匹配时返回上一个非空交集而非清空** |

## v5 修复说明（2026-05-17）

### Bug 描述

用户搜索 `"The Deinococcus radiodurans uvr"` 时，期望得到唯一结果 #1261，但 v4 算法返回了 30 个结果。

### 根本原因

v4 使用**子串匹配 + 投票计数**：
- `"uvr"` 作为子串匹配到了 `"uvrA1"`、`"UvrA2"`、`"UvrD"` 等，产生大量假阳性
- 投票后 #1037 和 #1261 并列最高（各 4 票），无法产生唯一解

### 测试数据

| 匹配方式 | `"uvr"` 匹配结果 | `"The Deinococcus radiodurans uvr"` 交集结果 |
|---------|------------------|---------------------------------------------|
| **子串匹配** | 6 篇（含 uvrA1, UvrA2, UvrD 等） | 2 篇（#1037, #1261） |
| **整词匹配** | **1 篇**（#1261: `"...uvr A gene..."`） | **1 篇**（#1261）✅ |

### 修复方案

将搜索算法从 **子串匹配 + 投票** 改为 **整词匹配 + 交集**：

1. **整词匹配函数**：
   ```javascript
   function containsWholeWord(title, word) {
       const regex = new RegExp('\\b' + word.toLowerCase() + '\\b', 'i');
       return regex.test(title);
   }
   ```
   - `\b` 匹配词边界（空格、标点、字符串开头/结尾）
   - 大小写不敏感
   - `"uvr"` 只匹配独立单词 `"uvr"`，不匹配 `"uvrA1"` 中的子串

2. **交集逻辑**：
   - 每个单词独立在全库检索整词匹配
   - 维护一个交集集合，初始为第 1 个单词的结果集
   - 第 2 个单词的结果集与当前交集取交集
   - 以此类推
   - 处理完前 8 个单词后，检查交集大小：
     - **为 1** → 返回该文献
     - **为 0** → 进入 fallback
     - **\>1** → 继续增加第 9 个单词

3. **渐进式处理**：
   ```
   单词1 → 结果集 S1
   单词2 → S1 ∩ S2 = I1
   ...
   单词8 → 检查 |I7| 是否 == 1
   不为1 → 单词9 → I7 ∩ S9 = I8，再检查
   ```

4. **Fallback 兜底**：
   - 整词交集为空 → 回退到**子串匹配**（v3 行为），返回包含所有查询单词子串的文献
   - 子串匹配也无结果 → fallback 链：作者搜索 → PMID → 摘要全文

### 边界处理

| 场景 | 处理方式 |
|------|---------|
| 查询少于 8 个单词 | 直接用全部单词计算交集 |
| 某个单词无整词匹配 | 交集为空，进入子串 fallback |
| 整词交集始终 \>1 | 返回交集内结果（最多 30 篇） |
| 整词交集为空 | 回退到子串匹配（所有单词必须都子串匹配） |
| 子串匹配也无结果 | 进入 fallback 链（作者 → PMID → 摘要） |

### 修改文件

| 文件 | 修改内容 |
|------|---------|
| `index.html` | `search()` 函数：子串投票 → 整词交集 + 子串 fallback |
| `app.py` | `api_search()` 函数：同上 |
| `搜索算法改进说明.md` | 添加 v5 修复记录 |

## v5.1 修复说明（2026-05-17）

### Bug 描述

用户搜索 `"The Deinococcus radiodurans uvr A ge xc"` 时无结果。

### 根本原因

v5 算法的 `buildIntersection()` 函数在构建交集时，**只在处理完所有单词后才检查唯一性**。当后续单词无匹配时，交集被清空，导致返回空结果。

具体流程（7个单词）：
1. `"The"` → 交集 ~561 篇
2. `"Deinococcus"` → 交集缩小
3. `"radiodurans"` → 交集缩小
4. `"uvr"` → 交集 = {1261}（**此时已唯一！**）
5. `"A"` → 交集仍为 {1261}
6. `"ge"` → 整词匹配 0 篇 → 交集被清空为 ∅
7. `"xc"` → 不再处理

**问题**：第4步已经得到唯一解 {1261}，但算法没有在第4步停止并返回，而是继续处理到第6步，被 `"ge"` 摧毁了交集。

### 修复方案

#### 修改 1：buildIntersection() — 每词检查后即时返回

- 引入 `lastNonEmpty` 变量记录上一个非空交集状态
- 当某个单词无整词匹配时，返回 `lastNonEmpty` 而非清空
- **关键修复**：每处理一个单词后立即检查交集是否唯一（size === 1），是则立即返回

#### 修改 2：逐轮增加块 — 同样添加 lastNonEmpty 保护

在 `words.length > BATCH_SIZE` 的 for 循环中：
- 添加 `let lastNonEmpty = resultSet.size > 0 ? new Set(resultSet) : null;`
- 当 `matches.size === 0` 时，返回 `lastNonEmpty` 而非清空
- 每次交集更新后记录 `lastNonEmpty`

#### 修改 3：同步修改 app.py

Python 版本的 `api_search()` 做完全相同的修复：
- 每处理一个单词后检查交集长度
- 长度为 1 时立即返回
- 单词无匹配时返回上一个非空交集

| `"The Deinococcus radiodurans uvr A ge xc"` | 空集 ❌ | 1篇 (#1261) ✅ |
